# Plan 04-03: CLI flag, startup validation, and tests

## Goal
Add `--max-slippage-bps` CLI flag to the `watch` command with default 50, validate at startup (reject 0 and >10000), and create integration tests covering the full slippage guard behavior.

## Context
Plans 04-01 and 04-02 built the slippage logic and wired it into the watch loop using `SlippageConfig::default()`. This plan completes the feature by adding the CLI flag so operators can configure the threshold, adding startup validation so bad values fail fast, and writing tests that prove the end-to-end behavior works: below-threshold allows rebalance, above-threshold aborts, and the CLI parses correctly.

   ```rust
       mint: String,
       #[arg(long, conflicts_with = "live")]
       shadow: bool,
       #[arg(long, conflicts_with = "shadow")]
       live: bool,
       /// Maximum allowed price impact in basis points (1 bps = 0.01%).
       /// Rebalances with simulated impact above this threshold are aborted.
       #[arg(long, default_value_t = 50)]
       max_slippage_bps: u32,
   },
   ```

2. **Add startup validation** in the `Commands::Watch` handler, before entering the main loop:
   ```rust

       anyhow::bail!(
           "--max-slippage-bps must be between 1 and 10000 (got {})",

       );
   }
   ```
   This must run BEFORE any async work or WebSocket connection.

3. **Replace `SlippageConfig::default()`** (added in Plan 04-02) with:
   ```rust
   let slippage_config = strategy::SlippageConfig { max_bps: *max_slippage_bps };
   ```
   This connects the CLI value to the slippage gate.

4. **Destructure the new field** in the match arm: update the Watch match arm to include `max_slippage_bps`.

Create `tests/slippage_tests.rs` with integration tests that exercise the public API of `strategy::slippage`:

1. **`test_small_trade_large_pool_passes_default_threshold`**:
   - Setup: price=150.0, liquidity=1_000_000_000_000 (1T -- realistic for major Orca pools), position_value=25_000.0 ($25k), config=default (50bps)
   - Assert: returns `SlippageResult::Ok { .. }` where `impact_bps < 50.0`

2. **`test_large_trade_tiny_pool_exceeds_threshold`**:
   - Setup: price=150.0, liquidity=1_000 (tiny pool), position_value=100_000.0, config=default
   - Assert: returns `SlippageResult::Abort { .. }` where `impact_bps > 50.0` and `threshold_bps == 50`

3. **`test_custom_threshold_changes_outcome`**:
   - Find a (price, liquidity, position_value) tuple where impact is around 20bps
   - With config max_bps=50: returns Ok
   - With config max_bps=10: returns Abort
   - This proves the threshold is respected, not hardcoded

4. **`test_zero_liquidity_always_aborts`**:
   - liquidity=0, any trade size > 0
   - Assert: returns Abort

5. **`test_zero_trade_size_always_passes`**:
   - position_value=0.0, any liquidity
   - Assert: returns Ok with impact_bps approximately 0.0

6. **`test_impact_increases_with_trade_size`**:
   - Same pool (price=100, liquidity=1_000_000_000), two trade sizes: $1000 and $50_000
   - Extract impact_bps from both results
   - Assert: larger trade has strictly higher impact_bps

7. **`test_cli_default_is_50_bps`**:
   - `assert_eq!(SlippageConfig::default().max_bps, 50);`

All tests use `tick_liq::strategy::slippage::*` imports (or `tick_liq::strategy::*` depending on re-exports). Check existing test files under `tests/` to match the crate name used in imports.
