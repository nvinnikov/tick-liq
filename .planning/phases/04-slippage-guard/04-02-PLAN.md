# Plan 04-02: Wire slippage gate into rebalance decision path

## Goal
Insert the slippage check between `should_rebalance()` returning `Rebalance` and `build_rebalance_plan()` being called in the watch loop, so that high-impact rebalances are aborted, logged, and persisted.

## Context
Plan 04-01 created the pure `check_slippage()` function. This plan wires it into the live watch loop in `src/main.rs`. Per CONTEXT.md decisions, the gate sits as a separate step between `should_rebalance()` and `build_rebalance_plan()`, mirroring the ShadowGuard separation-of-concerns pattern. On abort, a row is written to `shadow_rebalances` with `trigger_reason='slippage_abort'` and NULL simulated fields (matching the CONTEXT.md spec). The gate only fires when `should_rebalance()` returns `Rebalance` -- not on every tick.

```
should_rebalance() -> if Rebalance -> shadow guard check -> build_rebalance_plan() -> write shadow row
```

```
should_rebalance() -> if Rebalance -> slippage gate -> if Abort: log + write abort row + skip

```

Specific changes:

1. **Add SlippageConfig construction** near the top of the watch command handler (near where `rebalance_config` is created). Use `strategy::SlippageConfig::default()` for now -- CLI override comes in Plan 04-03. Store it in a variable accessible inside the tick callback closure.

2. **After the `is_rebalance` check (line ~638) and BEFORE the shadow guard submit + build_rebalance_plan block**, insert the slippage gate:

   ```rust
   // Slippage gate: compute price impact before any transaction path
   if is_rebalance {
       let position_value_usd = computed_position_value; // already available from line ~614
       let slippage_result = strategy::check_slippage(
           position_value_usd,
           price_current,
           pool.liquidity,
           &slippage_config,
       );
       match slippage_result {
           strategy::SlippageResult::Ok { impact_bps } => {
               tracing::info!(
                   impact_bps = impact_bps,
                   threshold_bps = slippage_config.max_bps,
                   position_value_usd = position_value_usd,
                   "slippage check passed"
               );
               // continue to shadow guard + build_rebalance_plan
           }
           strategy::SlippageResult::Abort { impact_bps, threshold_bps } => {
               tracing::warn!(
                   impact_bps = impact_bps,
                   threshold_bps = threshold_bps,
                   position_value_usd = position_value_usd,
                   "slippage guard: rebalance aborted — impact exceeds threshold"
               );
               // Write abort row to shadow_rebalances
               if let Some(ref pg) = db_pool {
                   let abort_row = storage::writer::ShadowRebalanceRow {
                       pool_address: pool_addr_clone.clone(),
                       trigger_reason: "slippage_abort".to_string(),
                       price: price_current,
                       simulated_range_width: None,
                       simulated_fees_earned: None,
                       simulated_il_usd: None,
                       simulated_net_pnl: None,
                       error_flag: false,
                       error_message: None,
                   };
                   storage::writer::spawn_shadow_write(pg.clone(), abort_row);
               }
               // Skip build_rebalance_plan entirely
           }
       }
   }
   ```

3. **Restructure the existing shadow row construction block** so that when slippage aborts, the normal `build_rebalance_plan()` path is skipped. Introduce a `let mut slippage_passed = false;` flag before the slippage check. Set it to `true` in the `SlippageResult::Ok` arm. Wrap the existing `build_rebalance_plan()` call and shadow row construction (lines ~648-708) in `if slippage_passed { ... }`.

4. **Use `computed_position_value`** (already computed around line 614) as the `position_value_usd` argument.

5. **Use `pool.liquidity`** as the liquidity argument (pool's active liquidity at the current tick, type u128).

Important constraints:
- Do NOT modify the `should_rebalance()` function or its signature
- Do NOT modify `RebalanceConfig`
- The gate runs ONLY when `is_rebalance` is true (not on every tick)
- The abort row uses `error_flag: false` (slippage abort is not an error -- it is protective behavior)
- The abort row uses `error_message: None`
- Slippage check runs in BOTH shadow and live modes (protects simulated decisions too, for logging fidelity)
