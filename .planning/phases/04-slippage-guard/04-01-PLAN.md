# Plan 04-01: Implement strategy::slippage module

## Goal
Create the `strategy::slippage` module with `SlippageConfig`, `SlippageResult` enum, and `check_slippage()` that computes price impact in bps for a given trade size and returns allow/abort.

## Context
Phase 4 protects capital from MEV and illiquid conditions by gating rebalances on simulated price impact. This plan builds the pure-logic core: given a position value (trade size proxy), current price, and pool liquidity, compute how many bps the trade would move the price, and compare against a configurable threshold. The existing `math::impact::estimate_impact()` computes USD needed to move price by X% -- this module inverts that relationship: given a USD trade size, find the resulting price impact in bps.

Create `src/strategy/slippage.rs` with:

1. **`SlippageConfig`** struct:
   ```rust
   pub struct SlippageConfig {
       pub max_bps: u32, // default: 50
   }
   ```
   Implement `Default` with `max_bps: 50`.

2. **`SlippageResult`** enum:
   ```rust
   pub enum SlippageResult {
       Ok { impact_bps: f64 },
       Abort { impact_bps: f64, threshold_bps: u32 },
   }
   ```

3. **`check_slippage()`** function:
   ```rust
   pub fn check_slippage(
       position_value_usd: f64,
       current_price: f64,
       liquidity: u128,
       config: &SlippageConfig,
   ) -> SlippageResult
   ```

   The function must:
   - Use binary search (or iterative refinement) over `math::impact::estimate_impact()` to find the `target_pct` whose `usd_needed` matches `position_value_usd`. This inverts the relationship: estimate_impact tells us "USD needed to move price X%", we need "what X% does our USD amount cause".
   - Convert the resulting `target_pct` to bps: `impact_bps = target_pct * 100.0` (since target_pct is in percentage points, 1.0 = 1% = 100bps).
   - Compare `impact_bps` against `config.max_bps as f64`.
   - Return `SlippageResult::Ok` if below, `SlippageResult::Abort` if at or above threshold.
   - Handle edge cases: if `liquidity == 0`, return `Abort` with `impact_bps = f64::INFINITY`. If `position_value_usd <= 0.0`, return `Ok` with `impact_bps = 0.0`.

   Binary search approach: search `target_pct` in range [0.001, 50.0] (0.001% to 50%). For each midpoint, call `estimate_impact(current_price, liquidity, mid, true)` and compare `.usd_needed` against `position_value_usd`. Converge when abs difference < 0.01 USD or after 50 iterations. Use `is_buy = true` as conservative estimate (buy-side impact is typically >= sell-side for token A).

4. **Unit tests** in `#[cfg(test)]` block within the same file:
   - `test_zero_liquidity_aborts`: liquidity=0 returns Abort with infinite impact.
   - `test_zero_trade_size_ok`: position_value_usd=0 returns Ok with 0 bps.
   - `test_small_trade_large_pool_ok`: $1000 trade against 10B liquidity at price 100 should be well under 50bps.
   - `test_large_trade_small_pool_aborts`: $100_000 trade against 1000 liquidity at price 100 should exceed 50bps.
   - `test_default_config_is_50_bps`: `SlippageConfig::default().max_bps == 50`.
   - `test_custom_threshold_respected`: set max_bps=10, verify a trade that passes at 50bps threshold now aborts.

5. **Update `src/strategy/mod.rs`**: Add `pub mod slippage;` and re-export: `pub use slippage::{check_slippage, SlippageConfig, SlippageResult};`
