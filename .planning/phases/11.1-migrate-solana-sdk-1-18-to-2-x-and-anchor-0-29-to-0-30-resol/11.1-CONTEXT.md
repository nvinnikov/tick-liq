# Phase 11.1: Solana SDK Migration — Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Поднять Solana Rust-стек с `solana-client`/`solana-sdk` 1.18 на **Solana 4.x**, закрыть 4 выдающихся CVE из `audit.toml` (curve25519-dalek, ed25519-dalek, quinn-proto, ring) и заменить сырой `tokio-tungstenite`-фид в `src/data/cex_ws.rs` на официальный `binance-sdk v45`. Edition = 2024, resolver = "3". Публичный контракт модуля `cex_ws` (`CexPriceState`, `watch_binance_price()`) и поведение `--cex-symbol` остаются без изменений.

**Важно:** ROADMAP success criterion #1 требует `anchor-client`/`anchor-lang 0.30+`, но Anchor в `Cargo.toml` отсутствует (вхождения «Anchor» в коде — только про 8-байтный discriminator borsh-аккаунтов, не framework). В этой фазе Anchor НЕ добавляется — scope переопределён (см. D-02). Success criterion #1 должен быть переписан после фазы.

</domain>

<decisions>
## Implementation Decisions

### Целевые версии
- **D-01:** `solana-client`/`solana-sdk` → **4.x** (latest: 4.0.1 от 17 фев 2026), НЕ 2.x из ROADMAP. Ecosystem за полтора года ушла на два мажора вперёд — мигрировать в промежуточный 2.x бессмысленно.
- **D-02:** **Anchor выходит из scope** — `anchor-client`/`anchor-lang` не добавляются. Причина: в текущем `Cargo.toml` их нет, прямых зависимостей у нас нет, а Anchor 1.0 тянет Solana 3.x → блокировал бы переход на 4.x. Если потребуется Drift CPI (LIVE-02, отложен) — добавим Anchor отдельной задачей, когда выйдет Anchor 2.x под Solana 4.x.
- **D-03:** `orca_whirlpools_core` остаётся на текущей мажорной (2.x) — крейт в своих deps не тянет `solana-*` (чистая CLMM math + wasm), миграция не влияет. Точную версию зафиксирует planner после `cargo update`.
- **D-04:** `binance-sdk v45` (15 апр 2026) — требует Rust ≥1.86 и tokio ≥1.33. У нас rustc 1.94.1, tokio "1" (full) — совместимо. TLS-бэкенд оставляем `openssl-tls` (default) если не упрёмся в `rustls-tls` на CI.
- **D-05:** `edition = "2024"`, `resolver = "3"` в `[package]` — переключаем одним commit в конце миграции (чтобы не смешивать с breaking API changes от SDK).

### Интеграция binance-sdk v45 (thin wrapper)
- **D-06:** Файл `src/data/cex_ws.rs` остаётся. Публичный API не меняется: `CexPriceState = Arc<RwLock<Option<CexPrice>>>`, `CexPrice { price: f64, updated_at: Instant }`, `async fn watch_binance_price(symbol: String, state: CexPriceState, shutdown: broadcast::Receiver<()>)`.
- **D-07:** Внутри функция вызывает `binance-sdk` — `spot::websocket::BookTicker` stream (или эквивалентный API v45, уточнит research). Reconnect/backoff/ping-pong отдаётся SDK, ручного `tokio-tungstenite`-loop больше нет.
- **D-08:** Unit-тесты модуля пересобираются:
  - **Удаляем** — `parse_book_ticker_*` (4 теста), `build_stream_url_lowercases_symbol`, `backoff_grows_then_resets`, `backoff_saturates_at_max` — дублируют функциональность SDK.
  - **Оставляем/переписываем** — `cex_price_struct_has_public_fields`, `handle_frame_updates_state`, `handle_frame_ignores_malformed_and_keeps_state` — проверяют наш адаптерный код (apply к state).
- **D-09:** 30-секундный staleness-fallback на on-chain `sqrt_price` в watch loop (`src/main.rs`) не трогаем — прежняя логика из Phase 11.
- **D-10:** `tokio-tungstenite` и `futures-util` остаются в `Cargo.toml` — их использует `src/data/ws.rs` (watch_account для Solana RPC WebSocket), не убираем.

### Порядок работ (атомарные commits в одной ветке)
- **D-11:** Один worktree/ветка, **4 атомарных commit**, каждый проходит `cargo build` + `cargo test` + `cargo clippy -D warnings` до мержа:
  1. `feat(11.1): migrate solana-sdk 1.18 → 4.x` — bump `solana-client`/`solana-sdk`, починить 5 файлов: `src/main.rs`, `src/rpc.rs`, `src/strategy/risk_monitor.rs`, `src/protocols/orca.rs`, `src/protocols/raydium.rs`.
  2. `feat(11.1): replace tokio-tungstenite CEX feed with binance-sdk v45` — переписать внутренность `src/data/cex_ws.rs`, подчистить тесты.
  3. `build(11.1): adopt edition 2024 and resolver 3` — флаги в `Cargo.toml`, любые правки под clippy.
  4. `chore(11.1): clear audit.toml ignore list` — удалить 4 старых advisories, добавить новые (если нужно) с обоснованием.
- **D-12:** Bisect должен работать по всей ветке — master/worktree остаётся всегда зелёным на каждом из 4 commits.

### Политика CVE
- **D-13:** Фаза зелёная если `cargo audit` на новом стеке: (a) исходные 4 advisories ушли; (b) любые новые критические/high — блокируют merge; (c) low/info-advisories допускаются с `ignore = [...]` в `audit.toml` + комментарий с RUSTSEC-ID, источником (какая transitive dep) и причиной допуска.
- **D-14:** Каждый новый `ignore` в `audit.toml` сопровождается ссылкой на upstream-issue или planned-fix, чтобы не превратить whitelist в свалку.

### Глубина research (перед planning)
- **D-15:** Лёгкий scout в `/gsd-plan-phase`: агент читает migration guide Solana 2→3→4, anchor-xyz/solana-sdk changelog, binance-sdk v45 docs и собирает в `11.1-RESEARCH.md` breaking changes по каждому из 5 touched src-файлов (симвлы, переехавшие в отдельные crates, removed APIs). Полный пофайловый анализ не нужен — compiler покажет остальное.

### Verification (что должно быть TRUE для закрытия фазы)
- **D-16:** `cargo build --release`, `cargo test`, `cargo clippy -- -D warnings` зелёные.
- **D-17:** `cargo audit` — 0 критических/high; исходные 4 RUSTSEC отсутствуют.
- **D-18:** Ручной smoke: `cargo run -- watch ... --cex-symbol SOLUSDT` в течение 5 минут — цена в `pnl_history.price` обновляется не реже раза в 5с и примерно совпадает с Binance mid.
- **D-19:** Все 350+ существующих unit-тестов проходят.

### Claude's Discretion
- Точная версия `orca_whirlpools_core` после `cargo update` (пока max совместимая 2.x).
- Имена в `binance-sdk` v45 API (Client vs Builder, Stream vs Handler) — уточнит research.
- Конкретные формулировки commit messages и clippy-правок.
- Решать ли на фронте `rustls-tls` vs `openssl-tls` (default openssl).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/data/ws.rs` — `watch_account()` с reconnect/ping-pong. После миграции остаётся на `tokio-tungstenite` 0.21 (или обновить до совместимого — research уточнит); это НЕ Binance-feed, его не трогаем.
- `src/data/cex_ws.rs` — thin wrapper остаётся, внутренность переписывается.
- `src/math/greeks.rs::sqrt_q64_to_price` — on-chain fallback когда CEX stale. Не трогается.

### Established Patterns
- Shutdown через `broadcast::Receiver<()>` — сохраняем, это внешний контракт `watch_binance_price`.
- `tracing::{info, warn}` — продолжаем использовать.
- Keypair/config через env — миграция этого не касается.
- Borsh-деериализация Anchor-аккаунтов с 8-байт discriminator (`src/protocols/orca.rs:21`) — логика остаётся, только `solana_sdk::pubkey::Pubkey` может переехать (см. `solana-pubkey` crate в 3.x+).

### Integration Points
- `src/main.rs` watch subcommand — вызывает `watch_binance_price` + `watch_account`; сигнатуры не меняются.
- `src/strategy/signal.rs` — тянет `tick_current: i32` и resolved-price; миграция этого не касается напрямую, но может быть транзитивный импорт `solana_*` типов.
- `src/rpc.rs` — RpcClient construction может потребовать правок (namespace changes в 2.x/3.x).

### Потенциальные breaking points (для research)
- `solana_sdk::pubkey::Pubkey` → в Solana 3.x часто вынесен в отдельный crate `solana-pubkey` и реэкспортируется.
- `solana_client::rpc_client::RpcClient` — интерфейс методов мог сузиться (deprecated методы могли быть удалены).
- Feature flags `solana-client` в 2.x+ перестроены (bpf/program/default).
- Программный account deserialize через `borsh` — зависит от версии `borsh` крейта, которую тянет solana-program.
- Зависимости `chrono`, `serde`, `thiserror` — проверить на совместимость с edition 2024.

</code_context>

<specifics>
## Specific Ideas

- "Solana 2.x — бессмысленный промежуточный шаг, берём сразу 4.x" — переопределение scope пользователем в ходе обсуждения.
- Цель повышения до Solana 4.x — не только закрыть 4 CVE, но и выйти на современную экосистему, чтобы следующие 1–2 мажора не требовали срочных миграций.
- Thin wrapper для `cex_ws.rs` предпочтителен — сохраняет границу абстракции и 30s staleness-логику в watch loop без рефакторинга main.rs.
- Bisect-совместимость — требование: любой commit из 4 должен быть зелёным и на build, и на test.

</specifics>

<deferred>
## Deferred Ideas

- **Добавление Anchor framework** — когда появится Anchor 2.x под Solana 4.x и/или возникнет реальная задача на Drift CPI (разблокировка LIVE-02). Сейчас нет ни зависимости, ни пользователя.
- **Замена `tokio-tungstenite` в `src/data/ws.rs`** — Solana 4.x может предложить нативный pubsub-клиент, который заменит ручной WS loop для `watch_account`. Не в scope этой фазы, отдельная задача.
- **Multi-exchange CEX fallback** (Kraken/Bybit) — унаследовано из deferred Phase 11, по-прежнему отложено.
- **rustls-tls backend для binance-sdk** — переключение с default openssl на rustls для более простой Docker/CI сборки. Не обязательно в этой фазе.
- **Переход на async sqlx 0.9 / tokio 2.x** — отдельная техмиграция, если появятся причины.

</deferred>
