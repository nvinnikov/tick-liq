## Fixed Issues

### WR-01: `decision_result` — `Result<_, String>` without an `Err` path

**Files modified:** `src/main.rs`
**Commit:** b801cb7
**Applied fix:** Replaced `let decision_result: Result<_, String> = Ok(should_rebalance(...))`
with a direct `let decision = should_rebalance(...)`. Updated both downstream
`match` sites (Telegram `trigger_reason` on 1130 and `shadow_row` on 1214) to
match `RebalanceDecision` variants directly. Removed the unreachable `Err(e)`
arm from `shadow_row` along with its `error_flag=true` row emission — that
branch was dead code since `should_rebalance` is infallible. Updated the inline
comment on 1092-1099 and 1211-1214 to document the decision, and left a note
that a future fallible/`catch_unwind` refactor should reintroduce the error arm.
Re-review flagged this as a regression of the Plan 03 "bad rows" invariant:
after this fix, no shadow-rebalance row ever has `error_flag=true` — which is
now *correct* because the code path cannot produce an error. If Plan 03 still
expects `error_flag=true` rows to exist for gating, that invariant needs to be
revisited separately (tracked as human-verification item).

Note: this commit also carries the IN-04 change because both findings touch
the same file (`src/main.rs`); see IN-04 below for the description.

### WR-02: `retry` unit tests sleep 750 ms / 250 ms due to `sleep_backoff` fallback

**Files modified:** `src/rpc.rs`
**Commit:** 4c39a8e
**Applied fix:** Replaced the single `const RETRY_BASE: Duration = Duration::from_millis(250)`
with a `#[cfg(not(test))] … = 250ms` / `#[cfg(test)] … = 0ms` pair
(lines 20-30 in rpc.rs). Production retry semantics (sequence 250 ms → 500 ms
→ 1 s across three attempts) are preserved; under `cargo test` the sync
`#[test]` functions now effectively no-op in `std::thread::sleep(Duration::ZERO)`.
Picked variant (1) from the reviewer's suggestion — parameterising `RETRY_BASE`
via cfg — since it was the simpler change and unit-test-sufficient; variant
(2) (tokio multi-thread test + `time::pause`) remains available if we later
need stricter determinism around the multi-thread runtime branch of
`sleep_backoff`.

### IN-01: `let _ = day_fees;` dead computation in backtest replay

**Files modified:** `src/backtest/db_replay.rs`
**Commit:** 9992fe5
**Applied fix:** Removed the dead `let day_fees = cumulative_fees - day_fees_at_start;`
computation (was on line 116), the paired `let _ = day_fees;` warning
suppression (was on line 139), and the now-unused `day_fees_at_start` accumulator

(total since replay start); left a comment at the day-boundary flush
documenting that per-day deltas were dropped and where to restore them if a
future `DayResult` schema exposes daily fee deltas explicitly.

### IN-02: Audit ignores without a concrete expiry milestone

**Files modified:** `.cargo/audit.toml`
**Commit:** f7803a2
**Applied fix:** Rewrote the `Expiry:` blocks for `RUSTSEC-2024-0370`

### IN-03: `let _ = shutdown.recv().await;` swallows `RecvError` in cex_ws

**Files modified:** `src/data/cex_ws.rs`
**Commit:** 4f638b4
**Applied fix:** Replaced the `let _ = shutdown.recv().await;` at line 100 with
an explicit `match` that logs `info!` on `Ok(())` (normal shutdown path) and
`warn!` on `Err(e)` (channel closed or receiver lagged). Both arms still fall
through to `connection.disconnect()` → `return`, preserving the previous
behaviour where a dropped shutdown sender caused a silent immediate exit —
now the unexpected closure is at least visible in logs. Added an inline
comment tying this to IN-03 and explaining that both arms mean "no more
shutdown signals will arrive → proceed to disconnect".

### IN-04: `warn!("cex_ws: feed task exited")` on graceful shutdown

**Files modified:** `src/main.rs`
**Commit:** b801cb7 (same commit as WR-01 — both findings touch `src/main.rs`)
**Applied fix:** Demoted the post-`watch_binance_price` log at line 766 from
`tracing::warn!("cex_ws: feed task exited")` to
`tracing::info!("cex_ws: feed task exited cleanly")` and added a three-line
comment documenting that `watch_binance_price` only returns on shutdown, so
the message fires on the graceful path and `warn!` was misleading. Because
this change lives in the same source file as WR-01, both landed in commit
`b801cb7`; the commit message names WR-01 (the bigger change), and this
report is the canonical record of the IN-04 fix being included.
