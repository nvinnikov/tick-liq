## Fixed Issues

### WR-01: Блокирующий `std::thread::sleep` внутри tokio-запускаемого retry

**Files modified:** `src/rpc.rs`
**Commit:** `8436ae8`
**Applied fix:** Заменил прямой `std::thread::sleep(delay)` в `SolanaRpc::retry` на новый helper `sleep_backoff(delay)`. Helper детектирует наличие multi-thread tokio runtime через `Handle::try_current()` + `runtime_flavor()`; если доступен, использует `block_in_place` + `tokio::time::sleep` (не блокирует worker — рантайм продолжает обслуживать другие задачи). Вне tokio-контекста (или на current-thread рантайме, где `block_in_place` недоступен) падает на обычный `std::thread::sleep`. Sync API и все call sites `fetch_account_checked` / `fetch_mint_decimals` не изменились, 10 существующих тестов `rpc::` прошли без правок.

### WR-02: `ConfigurationWebsocketStreams::builder().build()` без указания endpoint + отсутствие initial-connect retry

**Files modified:** `src/data/cex_ws.rs`
**Commit:** `c5f1bf8`
**Applied fix:** Вынёс шаги connect+subscribe в отдельную функцию `try_connect_and_subscribe()` и обернул её в reconnect-loop с backoff (`CONNECT_RETRY_DELAY = 5s`). Loop завершается одним из двух путей: (а) успешный connect → держим соединение до shutdown, затем `disconnect().await`; (б) любая transient-ошибка на старте → `tokio::select!` между sleep и shutdown, чтобы не делать лишнюю попытку прямо перед выходом. Initial failure теперь не "тихо" возвращает из watch_binance_price — feed восстановится после восстановления сети. Endpoint вопрос (`.ws_url`) оставлен на production-дефолте binance-sdk v45: `SpotWsStreams::production` явно присваивает `SPOT_WS_STREAMS_PROD_URL`, так что `.url()` не нужен.

Примечание про IN-03: pin на точную версию binance-sdk не входит в scope этого fix (info-level, пропущен по фильтру `critical_warning`).

### WR-03: `_cex_handle` / `_bot_handle` не ждутся и не мониторятся

**Files modified:** `src/main.rs`
**Commit:** `d460c5b`
**Applied fix:** Убрал `_` префикс у обоих JoinHandle (`bot_handle`, `cex_handle`). После `watch_account(...).await` добавлен cleanup-блок: cex-задача слушает тот же shutdown broadcast, поэтому `.await` её handle даёт graceful disconnect (новый reconnect-loop из WR-02 корректно обработает exit); бот не подписан на shutdown channel, поэтому сначала `abort()`, потом `.await` (ожидаемо отдаст `JoinError::cancelled`). clippy чистый, `cargo build` проходит без warnings. Помечено как "fixed" (не требует human verification), так как изменение локализовано в Watch-ветке и логика очевидна — await уже выполнившегося Future.
