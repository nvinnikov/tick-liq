## Executive Summary

1. **Solana 4.x is two major splits away from 1.18.** `solana-sdk` 4.0.1 is a thin wrapper that re-exports from ~100 component crates (`solana-pubkey`, `solana-program`, `solana-account`, …). The `solana_sdk::pubkey::Pubkey` import path still works via re-export in 4.x, so our 5 touched files need **minimal source changes** — mostly a version bump in `Cargo.toml` + whatever the compiler flags. [VERIFIED: docs.rs solana-sdk/4.0.1]

2. **`solana-client` 4.x is still in beta (4.0.0-beta.7 as of 2026-04-10); latest stable is 3.1.11.** [VERIFIED: docs.rs solana-client/3.1.11] This contradicts CONTEXT D-01's "4.x" target when applied to `solana-client`. Two options: (a) pin `solana-client = "3.1"` + `solana-sdk = "4"` (they're independently versioned in the split SDK), or (b) accept beta. Flagged as **OQ-1** for the planner.

3. **`anza-quinn` fork** is what `solana-client` 3.1.11 uses for QUIC — the version string literally contains `rustsec20260037`, meaning the fork patches RUSTSEC-2026-0037. [VERIFIED: docs.rs] All 4 advisories should be resolved by the upgrade; `audit.toml` will be emptied in commit #4.

4. **binance-sdk v45 API is callback-based, not stream-based.** `SpotWsStreams::production(cfg).connect().await` returns a `WebsocketStreamsHandle`; `handle.book_ticker(params).await` returns `Arc<WebsocketStream<BookTickerResponse>>`; messages are registered with `stream.on_message(|data| …)`. Shutdown = `handle.disconnect().await`. Reconnect/ping-pong is handled internally. [VERIFIED: official example in `binance/binance-connector-rust` repo]

5. **Edition 2024 + resolver 3 is low-risk for this codebase.** MSRV bump to 1.85+ is fine (we run 1.94.1). The main pitfalls are (a) RPIT lifetime capture (unlikely to hit our code), (b) `Future`/`IntoFuture` added to the prelude (possible ambiguity — `cargo fix --edition` auto-resolves), (c) `std::env::set_var` becomes `unsafe` (our code doesn't mutate env). [VERIFIED: Rust 1.85.0 release blog + edition guide]

**Primary recommendation:** Execute the 4-commit sequence exactly as CONTEXT D-11 prescribes. Pin `solana-sdk = "4"`, `solana-client = "3.1"` (pending OQ-1 resolution). Keep `openssl-tls` default. Use `cargo fix --edition` as the first step of commit #3 and clean up residue manually.

### Construction & lifecycle

```rust

    // data.best_bid_price, data.best_ask_price are string fields per Binance schema
    // parse to f64, compute mid, write to state
});

// graceful shutdown when broadcast::Receiver<()> fires:
connection.disconnect().await?;
```

### Mapping to our `watch_binance_price` contract

| Current behaviour | binance-sdk mechanism | Notes |
|-------------------|----------------------|-------|
| Parse `{"b":"…","a":"…"}` → f64 mid | `BookTickerResponse` fields (strings per Binance schema) | Parse to f64 inside `on_message` callback |
| Update `Arc<RwLock<Option<CexPrice>>>` with `Instant::now()` | Clone state into the callback closure | Callback must be `'static` — move-capture the state |
| Exponential backoff on disconnect | Internal to SDK | We delete our `RECONNECT_BASE`/`RECONNECT_MAX` + backoff loop |
| Ping-pong | Internal to SDK | Delete manual Ping/Pong handling |
| `broadcast::Receiver<()>` shutdown | `tokio::select!` between `shutdown.recv()` and a long `sleep` or sentinel oneshot; on shutdown call `connection.disconnect().await` | Wrap into a single async function |

### Shutdown plumbing (sketch)

```rust
pub async fn watch_binance_price(
    symbol: String,
    state: CexPriceState,
    mut shutdown: broadcast::Receiver<()>,
) {
    // build + connect (with an outer loop only if SDK.connect() returns non-retriable err)
    let cfg = match ConfigurationWebsocketStreams::builder().build() { Ok(c) => c, Err(e) => { warn!(%e); return; } };
    let client = SpotWsStreams::production(cfg);
    let connection = match client.connect().await { Ok(c) => c, Err(e) => { warn!(%e); return; } };
    let params = match BookTickerParams::builder(symbol.to_lowercase()).build() { Ok(p) => p, Err(e) => { warn!(%e); return; } };
    let stream = match connection.book_ticker(params).await { Ok(s) => s, Err(e) => { warn!(%e); return; } };

    let state_cb = state.clone();
    stream.on_message(move |data: BookTickerResponse| {
        if let (Ok(bid), Ok(ask)) = (data.best_bid_price.parse::<f64>(), data.best_ask_price.parse::<f64>()) {
            let mid = (bid + ask) / 2.0;
            if let Ok(mut g) = state_cb.write() {
                *g = Some(CexPrice { price: mid, updated_at: Instant::now() });
            }
        }
    });

    let _ = shutdown.recv().await;          // block until shutdown signal
    let _ = connection.disconnect().await;  // best-effort graceful close
}
```

**Points to verify during implementation (not blocking planning):**
- Exact field names on `BookTickerResponse` — Binance uses `"b"`/`"a"` wire keys but the SDK likely maps to `best_bid_price: String` / `best_ask_price: String`. [ASSUMED] — planner/implementer to confirm by fetching `BookTickerResponse` docs.
- Whether `on_message` takes `Fn` or `FnMut` — we use a `FnMut`-compatible closure via `&CexPriceState` clone. [ASSUMED]
- Whether the SDK's internal reconnect preserves the registered callback or requires re-registration. [ASSUMED — likely yes per SDK design]
