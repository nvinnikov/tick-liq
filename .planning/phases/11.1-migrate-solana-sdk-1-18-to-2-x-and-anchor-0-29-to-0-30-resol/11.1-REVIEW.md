## Warnings

### WR-01: `decision_result` — `Result<_, String>` без пути в `Err`

**File:** `src/main.rs:1093-1100` (+ обработка `Err(e)` на 1247-1264)
**Issue:** Конструкция

```rust
let decision_result: Result<strategy::RebalanceDecision, String> =
    Ok(strategy::should_rebalance(/* ... */));
```

жёстко захардкожена на `Ok(...)`. Ветка `Err(e)` в match на 1247-1264 (пишет строку в `shadow_rebalances` с `error_flag=true`) — мёртвый код: `decision_result` никогда не может быть `Err`, т.к. `should_rebalance` возвращает `RebalanceDecision`, а не `Result`. Комментарий на 1090-1091 обещает "wrap the full decision path so any error sets error_flag=true" — но тело `Ok(...)` без `try`/`?`/`map_err` не реализует эту защиту. На ре-ревью после миграции 11.1 это становится заметнее: код читается как будто миграция SDK могла внести panic/Err, а реально никакая ошибка не ловится.

**Fix:** Либо убрать `Result`-обёртку и упростить до

```rust
let decision = strategy::should_rebalance(/* ... */);
```

с прямым `match decision`, либо завернуть вызов в `std::panic::catch_unwind` / реально fallible path, если идея — ловить panics. В текущем виде error-ветка `shadow_rebalances` недостижима, что ломает инвариант «гейт считает bad rows» из Plan 03.

### WR-02: `retry`-тесты в `src/rpc.rs` используют `tokio::task::block_in_place` в sync `#[test]`

**File:** `src/rpc.rs:304-320` (`retry_exhausts_attempts_on_persistent_error`, `retry_succeeds_on_second_attempt`)
**Issue:** Тесты объявлены как `#[test]` (sync), но `retry()` после фикса WR-01 дергает `sleep_backoff()`, который проверяет `tokio::runtime::Handle::try_current()`. В sync-тесте handle отсутствует, поэтому ветка `MultiThread` не срабатывает, и тесты падают в `std::thread::sleep` — это работает, но делает тесты медленными: `retry_exhausts_attempts_on_persistent_error` теперь спит 250+500 = 750 ms (раньше был мгновенным до фикса; сейчас это tradeoff). Для sync-теста это терпимо, но если кто-то запустит suite с `--test-threads=1`, общее время заметно вырастёт. Также `retry_succeeds_on_second_attempt` спит 250 ms (одна итерация backoff перед успехом).

**Fix:** Два варианта:
1. Параметризовать `RETRY_BASE` (например, через `#[cfg(test)] const RETRY_BASE: Duration = Duration::from_millis(0)` или передавать как поле struct) — чтобы тесты не спали.
2. Переписать тесты как `#[tokio::test(flavor = "multi_thread")]` и использовать `tokio::time::pause()` / `advance()` для детерминизма.

Вариант 1 проще и достаточен для unit-уровня; вариант 2 «правильнее» архитектурно.

### IN-02: Отсутствующий expiry-дедлайн в `.cargo/audit.toml` для `RUSTSEC-2024-0370` / `RUSTSEC-2025-0134`

**File:** `.cargo/audit.toml:61, 77`

### IN-03: `_ = shutdown.recv().await` в `cex_ws.rs:100` теряет результат receive

**File:** `src/data/cex_ws.rs:100`
**Issue:** `let _ = shutdown.recv().await;` — игнорирует `Err(RecvError::Closed)` / `Err(RecvError::Lagged(n))`. Первый случай означает, что все `shutdown_tx` дропнуты (в `main.rs` это может случиться раньше выхода задачи), и тогда receive сразу вернёт `Err` без ожидания; задача выйдет без сообщения — но вызвавший код ожидает `cex_handle.await` и увидит нормальное завершение. Это функционально корректно, но если в будущем появятся дополнительные broadcast-ветки (например, reload config), молчаливое подавление `Err` скроет bugs.

**Fix:** Явно матчить и логировать:

```rust
match shutdown.recv().await {
    Ok(()) => info!("cex_ws: shutdown received, disconnecting"),
    Err(e) => warn!("cex_ws: shutdown channel closed unexpectedly: {}", e),
}
```

### IN-04: `cex_handle` теперь `Option<JoinHandle<()>>` c `tracing::warn!("cex_ws: feed task exited")` внутри spawn

**File:** `src/main.rs:764-770`
**Issue:** После фикса WR-03 `cex_handle.await` в shutdown path (1282-1286) ожидает завершения. Внутри spawn тело задачи:

```rust
data::cex_ws::watch_binance_price(...).await;
tracing::warn!("cex_ws: feed task exited");
```

`watch_binance_price` нормально возвращается только на shutdown (loop с `return`). В итоге **каждое** завершение — даже штатное по Ctrl+C — печатает `WARN cex_ws: feed task exited`. На ре-ревью это путает: warn предполагает «что-то пошло не так», а штатный shutdown → warn. Это регрессия UX ранее зафиксированного WR-03.

**Fix:** Понизить до `info!` или удалить строку (await в main.rs достаточен как сигнал «задача дожата»):

```rust
Some(tokio::spawn(async move {
    data::cex_ws::watch_binance_price(sym_owned, state_clone, cex_shutdown).await;
    tracing::info!("cex_ws: feed task exited cleanly");
}))
```
