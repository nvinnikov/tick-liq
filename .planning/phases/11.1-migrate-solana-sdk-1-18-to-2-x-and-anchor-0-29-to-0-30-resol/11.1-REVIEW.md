## Warnings

### WR-01: Блокирующий `std::thread::sleep` внутри tokio-запускаемого retry

**File:** `src/rpc.rs:62`
**Issue:** `retry()` использует `std::thread::sleep(delay)` для backoff. Метод `fetch_account_checked` вызывается из async context в `src/main.rs` (например, строки 535, 544, в tick-callback 798). Блокирующий `thread::sleep` на worker thread tokio рантайма (даже с `flavor = "multi_thread"`) останавливает поток, лишая рантайм возможности выполнять другие задачи. Во время RPC-retry задержек WebSocket-обработчики, Telegram dispatcher и другие taski могут получить stall. Это не новая проблема 11.1, но миграция на Solana 4.x обнажает её: retry стал критическим путём (x509 TLS handshake в новом `solana-client 4.0.0-beta.7` может вызывать больше transient errors).

**Fix:** Для sync-интерфейса оставить как есть, но для async path добавить async-совместимый вариант:
```rust
// Вариант 1: раскрыть асинхронный retry.
pub async fn retry_async<F, T, Fut>(&self, label: &str, mut f: F) -> Result<T>
where
    F: FnMut() -> Fut,
    Fut: std::future::Future<Output = Result<T>>,
{
    let mut last_err = anyhow!("{label}: no attempts made");
    for attempt in 0..MAX_ATTEMPTS {
        if attempt > 0 {
            let delay = RETRY_BASE * 2u32.pow(attempt - 1);
            tokio::time::sleep(delay).await;
        }
        match f().await {
            Ok(v) => return Ok(v),
            Err(e) => {
                tracing::warn!("{label}: attempt {attempt} failed: {e}");
                last_err = e;
            }
        }
    }
    Err(last_err)
}
```
Как минимум — завернуть sync-call в `tokio::task::spawn_blocking` на стороне caller, если нельзя трогать API.

### WR-02: `ConfigurationWebsocketStreams::builder().build()` без указания endpoint

**File:** `src/data/cex_ws.rs:74-80`
**Issue:** `ConfigurationWebsocketStreams::builder().build()` вызывается без указания `.url(...)` / `.ws_url(...)`. По binance-sdk v45 API конфиг может требовать явный WS endpoint — `SpotWsStreams::production(cfg)` ниже выбирает production endpoint, но полагаться на implicit default опасно: если upstream в минорной версии потребует `.url()` явно, `.build()` начнёт фейлиться, и функция тихо выйдет (предыдущий `tokio-tungstenite` код автоматически переподключался, новый — возвращается из функции на первой же ошибке). Нет также retry-петли: одна неудача `connect().await` / `book_ticker().await` = feed выключен до перезапуска процесса.

**Fix:** 
1. Проверить в docs binance-sdk v45 корректность default config для spot WS production.
2. Обернуть подключение в reconnect-loop (аналогично прежнему коду с exponential backoff), так как комментарий `/// Reconnect / ping-pong / backoff handled by binance-sdk v45` справедлив только для уже установленного соединения; initial-connect failure не реконнектится автоматически:
```rust
loop {
    if shutdown.try_recv().is_ok() { return; }
    match try_connect_and_subscribe(&symbol, &state).await {
        Ok(connection) => {
            let _ = shutdown.recv().await;
            let _ = connection.disconnect().await;
            return;
        }
        Err(e) => {
            warn!("cex_ws: connect failed: {e}; retrying in 5s");
            tokio::time::sleep(Duration::from_secs(5)).await;
        }
    }
}
```

### WR-03: `_cex_handle` / `_bot_handle` не ждутся и не мониторятся

**File:** `src/main.rs:698, 760`
**Issue:** `let _bot_handle: Option<JoinHandle<()>> = ...` и `let _cex_handle = ...` создают tokio-задачи, но handle хранится с префиксом `_`, что запрещает компилятору жаловаться на unused. При этом в конце `Watch` ветки (`data::ws::watch_account(...).await`) возврат/panic этих задач не обрабатывается: если Binance feed падает из-за WR-02, это видно только через `tracing::warn!("cex_ws: feed task exited")` из inner spawn, но основной loop продолжает работать с on-chain ценой как fallback. Это частично by design (fallback на on-chain price), но отсутствие явного `.await`/`.abort()` на shutdown означает, что при выходе из `watch_account` оба handle сразу дропаются, а задачи получают abort без graceful cleanup. Для bot это может привести к tг update loop без shutdown (задача живёт, пока runtime умирает).

Похожая проблема уже исправлена в phase 11 для cex feed (коммит `ca5d955` — "retain cex feed JoinHandle to log unexpected task exit"), но здесь переменная по-прежнему префиксована `_`, то есть mpv-warning "unused" подавляется формально.

**Fix:** Убрать `_` из имени и при shutdown явно дождаться:
```rust
let bot_handle: Option<tokio::task::JoinHandle<()>> = ...;
let cex_handle = ...;

// в конце Watch:
data::ws::watch_account(ws_url, pool_addr, shutdown_rx, on_notify).await;

if let Some(h) = cex_handle {
    let _ = h.await; // даст taski gracefully завершиться после broadcast
}
if let Some(h) = bot_handle {
    h.abort(); // bot не слушает shutdown — abort после выхода
    let _ = h.await;
}
```

### IN-01: Double-negation в `net_pnl` вычислении

**File:** `src/main.rs:965, 1138, 1237`
**Issue:** `computed_fees_earned - computed_il_usd.abs()` используется вместо `computed_fees_earned + computed_il_usd`. `compute_il()` возвращает неположительное значение (IL по контракту ≤ 0), поэтому `fees + il` эквивалентно `fees - |il|`. Две разные формы соседствуют в одном файле (строка 1098 использует `computed_fees_earned + computed_il_usd`, строки 965/1138/1237 — `- .abs()`), что создаёт риск рассинхрона, если знак `compute_il` будет когда-нибудь изменён.

**Fix:** Привести к одной форме — `computed_fees_earned + computed_il_usd`. Если инварианта `il <= 0` не хватает, сделать `debug_assert!(il_usd <= 0.0)` в `compute_il`.

### IN-02: `fee_apy_pct` использует `current_day_num` вместо реального диапазона дат

**File:** `src/backtest/db_replay.rs:217`
**Issue:** `total_days` считается как `current_day_num` — счётчик дней в данных. Если тики разрежены (например, 3 тика во второй и восьмой календарный день), `current_day_num = 2`, но реальный временной диапазон — 6 дней. `fee_apy_pct = (fees/capital) * 365/total_days * 100` даст завышенный APY в 3 раза.

**Fix:** Считать реальное количество календарных дней через `(ticks.last().time.date_naive() - ticks[0].time.date_naive()).num_days() + 1`:
```rust
let total_days_real = (last.time.date_naive() - ticks[0].time.date_naive()).num_days().max(1) as u32;
let fee_apy_pct = if input.initial_value_usd > 0.0 {
    (cumulative_fees / input.initial_value_usd) * (365.0 / total_days_real as f64) * 100.0
} else { 0.0 };
```

Примечание: это не регрессия 11.1, файл был затронут только в test-only местах (`_u128 as f64` → `_f64`). Но поскольку фикс мелкий и fee APY в DB-режиме — ключевая операторская метрика, отмечаю здесь.

### IN-03: `binance-sdk` зависимость без `default-features = false` pin к минорной версии

**File:** `Cargo.toml:41`
**Issue:** `binance-sdk = { version = "45", features = ["spot"], default-features = false }` использует `^45` (any 45.x.y). Upstream SDK — beta-level продукт (релиз 2026-04-15), API поля могут меняться в minor-версиях. В частности `BookTickerResponse.b/a` типы (`Option<String>`) могут стать strict `String` в 45.1. Комментарий в коде упоминает конкретную версию 45.0.0, но Cargo.toml этого не закрепляет.

**Fix:** Закрепить точнее — `version = "=45.0.0"` (exact) или `"45.0"` (tilde). Альтернативно, подтвердить в comment что caret-range приемлем с тест-покрытием API.
